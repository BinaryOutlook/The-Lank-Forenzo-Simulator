# V0.3.1 PRD

Status: Implemented
Owner: BinaryOutlook
Last updated: 2026-04-10
Iteration type: Decision-library data augmentation patch

## 1. Summary

`V0.3.1` is a patch-scale follow-up to `V0.3`.

Its purpose is simple and explicit:

- bring the authored decision library to `100` total decisions
- bring the pack structure to `10` total decision packs
- do it without a major code rewrite

This iteration is not a systems redesign. It is a content augmentation pass.

The main product value is scale, coverage, and maintainability. The game should feel broader, more replayable, and more historically textured, while the underlying simulation architecture remains intact.

## 2. Problem Statement

`V0.3` established the correct structure for a large authored decision library, but it stopped short of a clean milestone.

Three follow-up needs remained:

- The library had not yet reached the clearer long-term benchmark of `100` authored decisions.
- The pack structure was still at `9` packs, which left one obvious expansion lane underrepresented.
- Oversight-management, hearing optics, consent-order theater, and regulatory camouflage were present in tone, but not yet strong enough as a first-class decision family.

The result was a library that was already much better than the original baseline, but still had room for one more focused authored expansion pass.

## 3. Goals

This iteration should:

- increase total authored decisions from `90` to `100`
- increase total decision packs from `9` to `10`
- add one new pack with a clear tactical identity instead of scattering `10` disconnected cards across the library
- preserve the current content-loading, validation, tray-curation, and resolution architecture
- keep implementation primarily in authored data files, with only minor registry, schema, test, and documentation updates where needed

## 4. Non-Goals

This iteration does not:

- redesign the run-screen UI
- introduce a new simulation subsystem
- rewrite tray curation logic in a major way
- change the round structure
- add backend services, procedural content generation, or live delivery
- attempt a balance overhaul across the whole game

This is intentionally not a rewrite.

## 6. Requirements

### 6.1 Exact content target

Requirements:

- `V0.3.1` must land at exactly `100` total authored decisions.
- `V0.3.1` must land at exactly `10` total decision packs.
- The new total should come from additive authored content, not from padding with duplicate or low-value filler.

### 6.2 Data-first implementation

Requirements:

- The iteration should be primarily a data augmentation pass.
- Existing decision and event JSON authoring patterns should be reused.
- Existing loading, validation, and deterministic resolution systems should remain the foundation.
- TypeScript changes should stay limited to the minimum needed to register the new pack, validate it, and test it.

Implementation guidance:

- Prefer extending the current pack-based content model rather than introducing a new authoring abstraction.
- Avoid structural changes unless they are required to keep the new pack first-class.

### 6.3 New pack identity

The new pack for `V0.3.1` should be `regulatory_theater`.

Requirements:

- The pack should have a coherent identity instead of a grab bag of leftover ideas.
- It should cover tactics such as:
  - revolving-door hires
  - independent-review theater
  - hearing preparation and transcript optics
  - consent-order delay
  - inspector management
  - civic jobs messaging
  - visible concessions that hide deeper damage
  - internal reform structures used as reputation armor
- The pack should feel historically informed, but still fictionalized and reusable.

### 6.4 Consequence support

Requirements:

- The new pack should not feel consequence-light.
- High-risk or high-optics cards should point to delayed fallout.
- Supporting event additions are acceptable where existing event pools are too generic to carry the new pack.
- No new consequence system should be introduced just for this patch.

### 6.5 Maintainability

Requirements:

- The repo should remain easy to re-enter after the patch lands.
- Content totals in tests and docs should match reality.
- The new packet should be documented as a patch-scale augmentation, not as a major platform shift.

## 7. UX Acceptance Criteria

The iteration is successful when all of the following are true:

- The content library clearly reaches `100` decisions and `10` packs.
- The new pack feels like a real strategic lane, not a novelty appendix.
- The decision tray gains more regulatory and oversight-facing texture without changing its overall interaction model.
- The product still feels like the same game, only broader and sharper.
- Future content authors can keep extending the pack model without needing a new workflow.
