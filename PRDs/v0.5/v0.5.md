# V0.5 PRD

Status: Draft
Owner: BinaryOutlook
Last updated: 2026-04-17
Iteration type: Systems expansion, reachability repair, and systemic replayability foundation
Source brief: `idea.md`

## 1. Summary

`V0.5` is the first major systems-expansion iteration for **The Lank Forenzo Simulator**.

The current build is already playable, deterministic, local-first, content-rich, and well documented. It has a strong authored loop: board packet, curated decision tray, deterministic round resolution, operating drift, delayed and ambient events, ending checks, and local save persistence.

The problem is no longer "can the game work?" The problem is "can the game stay deep, varied, and maintainable as the possibility space expands?"

The master expansion brief in `idea.md` identifies the real bottleneck:

- the current game has strong editorial control but shallow systemic emergence
- the balance report collapses into extraction endings too reliably
- only a small share of authored decisions and events surface under seeded play
- factions exist mostly as metrics and authored consequences, not persistent actors
- delayed events are still closer to a list than a true event scheduler
- operations do not yet have an internal world model that can create emergent fallout
- run history is evocative but not yet a living scandal dossier

`V0.5` should convert the game from a sharp authored card-driven macro sim into a deeper deterministic simulation platform.

This iteration should land the first production-grade versions of:

- a stronger simulation boundary and compiled-content manifest
- reachability and balance tooling that can expose dominance and dead content
- constraint-aware tray composition that optimizes the whole tray, not just individual cards
- a scheduler that supports guaranteed consequences and hazard-driven pressure
- faction memory and deterministic faction intents
- a compact operational network layer for maintenance, weather, and route fragility
- a dossier and post-run scandal layer that turns history into evidence
- UI and persistence upgrades needed to make these systems legible without losing the current product identity

`V0.5` is a major iteration. It should be implemented in phases, with validation checkpoints between each phase. It is not a license to rewrite the app casually. The goal is to add systemic machinery while preserving determinism, local-first playability, and the boardroom-extraction fantasy.

## 2. Product Decision

The stable product thesis remains unchanged:

**The company and the player are not the same entity.**

`V0.5` expands the world around that thesis. The player should no longer feel that they are only choosing from a large authored deck. They should feel that factions, operations, scandal threads, and market narratives are remembering the style of their misconduct.

The player is still acting at the executive level. They are not dispatching flights, assigning crews, or role-playing a lawyer. They are making ugly strategic decisions and watching institutional, operational, and evidentiary systems respond.

The intended player-facing shift:

- Before `V0.5`: "I chose a card, metrics moved, and events answered later."
- After `V0.5`: "The way I keep winning is teaching the world how to come for me."

## 3. Assumptions

This PRD makes the following assumptions. If any assumption changes, update this PRD before implementation continues.

- `V0.5` remains a browser-native React and TypeScript app.
- The core game remains playable offline with no required backend.
- Runtime simulation remains deterministic for a given seed, content version, and selected decision sequence.
- The current package manager and Vite app stay in place unless a separate approved migration task is created.
- A full `pnpm` plus `Turborepo` workspace is a future target, not a default requirement for this iteration.
- Runtime LLM behavior is out of scope. AI-like behavior means deterministic faction planning, not a black-box model.
- The operational model is an executive-level substrate, not a detailed route-planning game.
- IndexedDB is allowed for run archives and replay-sized payloads if the implementation needs it, but tiny preferences can remain in `localStorage`.
- New content must remain fictionalized and satirical. It can be inspired by airline and corporate history, but it must not become a documentary reenactment.

## 5. Problem Statement

The current game has the right identity but not yet enough systemic depth.

The authored content library is large enough that repeated runs should be exposing many different strategic branches. Instead, seeded play currently collapses into a narrow extraction corridor. That means the issue is not simply lack of content. The issue is that the runtime curation, reachability, event timing, ending pressure, and world-memory systems are not yet forcing the content library into a broad enough possibility space.

The main problems are:

- Decision curation scores cards mostly as independent local choices instead of composing a round-level strategic tray.
- Event timing is too limited for future faction incidents, hearing calendars, operational cascades, and scandal chains.
- Factions are represented indirectly through metrics, flags, and events, but they do not yet maintain enough memory to adapt to player behavior.
- Operating drift is legible but too compressed. It does not yet have enough topology for systemic propagation.
- There is no operational substrate. Safety, workforce, route, weather, and maintenance pressure do not yet interact through a world model.
- History is presentation memory, not evidentiary memory. The game cannot yet explain a run as a scandal biography.
- Tooling can report balance symptoms, but cannot yet explore reachability, dead zones, or chain lock-in.

`V0.5` must address these as connected problems. A smarter tray without better reachability tooling is guesswork. Factions without a scheduler become noisy event flavor. An operational layer without UI discipline turns the product into a spreadsheet. A dossier layer without persistence and post-run presentation becomes hidden state no one appreciates.

## 6. Design Stance

`V0.5` should be ambitious but disciplined.

Guiding principles:

- Preserve the current game loop as the player's mental model.
- Add systemic depth behind the loop, not UI clutter in front of it.
- Keep deterministic execution as a hard product constraint.
- Prefer typed, inspectable state over clever implicit behavior.
- Use authored content as the voice of the game, but let systems decide timing, pressure, and consequence.
- Introduce new state only when it produces player-facing consequence or better tooling.
- Make balance problems visible with reports before tuning them by feel.
- Keep every new system small enough to test outside React.
- Treat `V0.5` as a platform upgrade for future versions, not a one-off feature dump.

The correct result is a game that feels more alive, not more random.

## 7. Goals

`V0.5` should:

- reduce dominant extraction convergence under seeded campaigns
- materially increase surfaced decision and triggered event coverage
- make decision trays more diverse and strategically tense
- introduce persistent faction memory for board, creditors, labor, regulators, and press
- let faction state influence events, decision availability, board packets, and post-run explanation
- replace list-like delayed-event handling with a scheduler interface that supports guaranteed and hazard events
- introduce a compact route and operations model that creates maintenance, weather, and network-fragility consequences
- introduce dossier threads so repeated misconduct creates distinct scandal trajectories
- add post-run analytics that explain why a run ended and what nearly happened
- keep the app local-first, deterministic, testable, and maintainable
- update docs, commands, and validation scripts so future agents can re-enter quickly

## 8. Non-Goals

`V0.5` does not:

- add a mandatory backend, account system, cloud save, or live-service dependency
- replace React, Vite, Zustand, Zod, Vitest, or Playwright
- migrate to a monorepo by default
- implement runtime LLM narrative generation
- become a detailed airline route-planning simulator
- simulate real-world airlines, disasters, executives, or legal cases one-to-one
- require photorealistic maps, 3D scenes, or a new landing page
- remove existing endings unless a documented balance decision requires it
- loosen deterministic tests to make new systems easier to land
- treat content volume alone as success

## 9. Success Metrics

The exact thresholds can be tuned during implementation, but `V0.5` should define and report against these targets.

Gameplay balance targets:

- no single ending exceeds `60%` across the default seeded scenario matrix
- extraction remains viable but no longer appears as the only obvious success corridor
- at least `55%` of authored decisions surface across the default matrix
- at least `35%` of authored events trigger across the default matrix
- every major decision pack surfaces under at least two scripted archetypes
- no strategic flag remains producer-only unless explicitly documented as future scaffolding

Tray composition targets:

- a normal tray should prefer at least `3` distinct tactic groups when the eligible pool allows it
- exact previous-round repeats should be suppressed unless the eligible pool is too small
- live follow-up decisions should surface within a bounded number of eligible rounds
- exit cards should not be starved when their requirements are truly met

Systemic depth targets:

- at least five factions persist state across rounds
- at least three faction intents can produce observable player-facing consequences
- at least one faction can adapt differently based on repeated player behavior
- at least one operational cascade can be traced from executive decision to network stress to macro metric impact
- at least three dossier themes can accumulate evidence and influence ending flavor or legal pressure

Technical targets:

- `npm run check` passes
- `npm run content:validate` passes with no unexplained warnings
- `npm run build` passes
- Playwright covers start, decision selection, round resolution, ending, and post-run recap
- save migration tests cover the pre-`V0.5` state shape and the `V0.5` state shape
- long-run analysis can run outside React
- browser UI remains responsive during normal play

## 10. Player Experience Requirements

### 10.1 Run loop

The core round rhythm should remain familiar:

1. Read the board packet.
2. Inspect metrics and pressure.
3. Choose a limited number of decisions.
4. Resolve the quarter.
5. Read consequences.
6. Watch new pressures and exit windows emerge.

`V0.5` should deepen what happens inside that loop.

The player should see:

- clearer explanations of why a faction is escalating
- operational warnings that feel like board-level intelligence, not dispatch detail
- consequence feed entries that reference faction, operational, or dossier causes
- post-run analysis that explains which style of play created which failure or escape

### 10.2 Strategic identities

The game should support several viable or semi-viable executive archetypes:

- merger groomer
- offshore operator
- denial machine
- restructuring cannibal
- regulatory theater specialist
- creditor trench fighter
- labor crusher
- reluctant stabilizer

These archetypes do not need formal character classes in `V0.5`. The systems should simply make them legible through repeated choices, faction responses, and post-run summaries.

### 10.3 Feedback quality

When the world reacts, the player should understand the cause.

Good feedback:

- "Labor organizing accelerated after two consecutive crew cuts and a contractor safety incident."
- "Regulators changed posture because the reform office flag is live but safety integrity kept falling."
- "The Nassau window narrowed because offshore readiness improved while press intensity also rose."
- "The route network absorbed the first weather shock, but maintenance backlog made the second one visible."

Bad feedback:

- a random-seeming event with no connection to recent play
- hidden faction state that changes outcomes with no explanation
- operational math that makes the player feel punished by a spreadsheet they cannot see

## 12. System Pillars

### 12.1 Simulation boundary and content manifest

`V0.5` should make the simulation core easier to call from the browser, tests, scripts, and future workers.

Requirements:

- Provide a single public simulation API for initializing runs, curating trays, resolving rounds, and summarizing run state.
- Keep React, Zustand, DOM APIs, and browser storage out of pure simulation modules.
- Create a compiled-content manifest or manifest-like runtime index.
- The manifest must include direct lookup maps for decisions, events, endings, flags, packs, and tags.
- Runtime resolution should avoid repeated linear lookups for delayed event ids.
- Save payloads should be able to pin a content version label or hash.
- Content diagnostics should report producer-only flags, consumer-only flags, unreferenced delayed events, likely impossible requirements, and pack coverage.

Recommended manifest shape:

```ts
interface CompiledContentManifest {
  version: string;
  contentHash: string;
  decisions: DecisionDefinition[];
  events: EventDefinition[];
  endings: EndingDefinition[];
  decisionById: Record<string, DecisionDefinition>;
  eventById: Record<string, EventDefinition>;
  endingById: Record<string, EndingDefinition>;
  decisionsByPack: Record<string, string[]>;
  eventsByTag: Record<string, string[]>;
  flagProducers: Record<string, string[]>;
  flagConsumers: Record<string, string[]>;
  diagnostics: ContentDiagnostic[];
}
```

Implementation guidance:

- Prefer an internal manifest module before introducing a separate package.
- If a compile step is added, make it explicit and testable.
- Avoid runtime fetches for local JSON content.
- Preserve content authoring friendliness.

Acceptance criteria:

- `loadContent` or its successor returns indexed content.
- `resolveRound` and tooling use `eventById` for event lookup.
- content validation reports the existing unused flag and unreferenced delayed-event issues.
- every report includes the manifest version or hash used.
- tests prove duplicate ids, broken references, and impossible requirements are still caught.

### 12.2 Reachability explorer and balance matrix

`V0.5` must upgrade balance tooling from summary reporting to exploratory diagnostics.

Requirements:

- Add an archetype-aware seeded simulation matrix.
- Add a reachability explorer that searches through decision combinations without pretending to solve the whole game.
- Track surfaced decisions, selected decisions, emitted events, endings, pack coverage, flag coverage, repeated tray pressure, and state dead zones.
- Use deterministic seeds and content hashes in every report.
- Output human-readable console summaries and machine-readable artifacts.
- Make thresholds visible but initially configurable.

Recommended report fields:

- content hash
- seed family
- run count
- max rounds
- bot policy or archetype
- ending distribution
- average run length
- decision surfaced count and percentage
- decision selected count and percentage
- event triggered count and percentage
- delayed-event trigger count
- hazard-event trigger count
- pack coverage
- faction-intent coverage
- dossier-theme coverage
- operational-cascade count
- repeated-tray pressure
- unreachable or low-confidence content ids

Suggested commands:

```bash
npm run simulate:runs
npm run balance:report
node --import tsx scripts/reachability-report.ts --width 32 --depth 24 --seed v0.5-default
node --import tsx scripts/balance-matrix.ts --runs 200 --max-rounds 24 --seed v0.5-matrix
```

Implementation guidance:

- The first explorer can use beam search over abstracted states.
- State abstraction should bucket metrics rather than keying every exact number.
- The tool should label confidence honestly.
- Use the explorer to find design blind spots, not to define optimal play.

Acceptance criteria:

- the tool can detect the current extraction-dominance baseline
- report output identifies low-reachability decision packs
- report output identifies low-reachability event families
- report output can be compared across content hashes
- unit tests cover state abstraction and novelty scoring

### 12.3 Constraint-aware tray composer

The decision tray should become a set-level composition problem.

The current system ranks decisions and then applies variety constraints. `V0.5` should optimize the tray as a whole. The objective is not just to show individually strong cards. It is to show a strategically tense set of choices.

The tray objective should consider \(F(T \mid s) = utility + diversity + chain urgency + counter-pressure - redundancy\).

Requirements:

- Keep the public caller surface close to the current `getAvailableDecisions` shape where practical.
- Preserve deterministic behavior for the same run state and seed.
- Keep hard eligibility requirements separate from soft scoring.
- Preserve exit-card availability when endings are live.
- Reward tactic diversity across the tray.
- Reward surfacing follow-ups for live playbooks.
- Penalize exact repeats from the previous tray.
- Penalize overrepresented packs across recent rounds.
- Surface counter-pressure when one playbook is dominating.
- Degrade gracefully when the eligible pool is too small.

Recommended inputs:

```ts
interface TrayContext {
  run: RunState;
  manifest: CompiledContentManifest;
  previousTrayIds: Set<string>;
  recentPackPressure: Record<string, number>;
  livePlaybooks: Set<string>;
  factionPressure: Partial<Record<FactionId, number>>;
  dossierThemes: Set<DossierTheme>;
}
```

Acceptance criteria:

- tray tests cover diversity, follow-up surfacing, exit-card preservation, repeat suppression, and small-pool fallback
- seeded matrix shows materially improved decision coverage over the `21%` baseline
- the composer produces deterministic output for fixed seeds and states
- tuning constants live in one inspectable module
- the composer reports enough diagnostics for balance tooling to explain why a tray was built

### 12.4 Event scheduler and hazard system

The current delayed-event array should evolve into a scheduler interface.

`V0.5` does not need an elaborate calendar UI. It needs a deterministic scheduling substrate that can support future hearing dates, faction threats, maintenance cascades, and scandal escalations.

Requirements:

- Represent guaranteed future events separately from hazard-driven event families.
- Use direct event lookup maps.
- Support cooldowns for hazard rules.
- Support stale windows so old consequences can expire cleanly.
- Support per-turn event budgets.
- Preserve authored delayed consequences.
- Migrate existing pending delayed events into the new state shape.
- Keep event resolution order deterministic and documented.

Recommended types:

```ts
interface ScheduledEvent {
  id: string;
  kind: "guaranteed" | "hazard";
  eventId: string;
  triggerRound: number;
  priority: number;
  staleAfterRound?: number;
  sourceRefs: string[];
}

interface HazardRule {
  id: string;
  eventId: string;
  baseWeight: number;
  cooldownRounds: number;
  tags: string[];
}

interface EventSchedulerState {
  queue: ScheduledEvent[];
  cooldowns: Record<string, number>;
  firedEventIds: Record<string, number>;
}
```

Event budget:

- at most one major scheduled event per round unless a forced ending is triggered
- at most one hazard event per round in the initial implementation
- unlimited hidden state updates if they are not shown as feed entries
- never drop guaranteed events silently; defer or mark stale with diagnostics

Acceptance criteria:

- all existing delayed events still resolve when their requirements are met
- hazard events are deterministic for fixed seed and run state
- cooldowns prevent the same hazard family from spamming the feed
- save migration preserves pending consequences from old saves
- tests cover due events, stale events, cooldowns, requirement failure, and event budgets

### 12.5 Factional political economy v1

Factions should become persistent actors with memory and pressure.

The goal is not diplomacy. The goal is institutional memory. The game should remember whether the player is abusing labor, deceiving creditors, performing regulatory theater, gaming press cycles, or grooming a board-approved exit.

Required factions:

- `board`
- `creditors`
- `labor`
- `regulators`
- `press`

Optional stretch faction:

- `politicians`

Recommended state:

```ts
type FactionId =
  | "board"
  | "creditors"
  | "labor"
  | "regulators"
  | "press"
  | "politicians";

interface FactionState {
  id: FactionId;
  patience: number;
  aggression: number;
  trust: number;
  cohesion: number;
  leverage: number;
  dossierWeight: number;
  recentGrievances: string[];
  lastIntentId?: string;
}
```

Required intent families:

- `pressure`
- `negotiate`
- `leak`
- `investigate`
- `organize`
- `defect`
- `shield`
- `replace`

Requirements:

- Faction state persists across rounds.
- Faction updates derive from selected decisions, emitted events, metrics, flags, and dossiers.
- At least three factions can emit intents.
- Intents can affect weighted event candidates, decision locks or boosts, metric pressure, board packet copy, or ending flavor.
- Faction behavior remains deterministic and inspectable.
- Faction state must not replace the visible metric vector. It should explain and deepen it.

Example integration:

```ts
interface FactionResolution {
  factions: Record<FactionId, FactionState>;
  intents: FactionIntent[];
  metricImpacts: Partial<RunMetrics>;
  scheduledEvents: ScheduledEvent[];
  decisionModifiers: DecisionModifier[];
  briefingSignals: BriefingSignal[];
}
```

Acceptance criteria:

- repeated labor damage increases labor grievance memory and can trigger labor-specific consequences
- repeated regulatory theater changes regulator behavior if safety or legal pressure keeps worsening
- board trust can diverge from market confidence
- creditor patience remains visible but gains deeper creditor-state causes
- the board packet can explain at least one faction intent in player-readable language
- unit tests cover faction update rules and deterministic intent selection

### 12.6 Operational world model v1

`V0.5` should add a compact operational substrate without turning the game into a route planner.

The operational layer exists to convert executive choices into credible operational fallout:

- deferred maintenance
- thin staffing
- contractor dependence
- hub consolidation
- route fragility
- weather shocks
- network disruptions

Recommended state:

```ts
interface NetworkState {
  hubs: HubState[];
  routes: RouteState[];
  fleet: FleetState[];
  crewPools: CrewPoolState[];
  weatherFronts: WeatherFrontState[];
  maintenanceBacklog: number;
}
```

Initial scale:

- `8` to `12` strategic hubs
- `12` to `24` route edges
- `2` to `4` fleet groups
- `2` to `4` crew pools
- `1` active weather-front model at a time

Requirements:

- The player does not manually assign flights.
- Executive decisions alter operational state indirectly.
- Operational state resolves once per round into macro impacts and possible events.
- Operational consequences affect existing metrics such as `airlineCash`, `safetyIntegrity`, `publicAnger`, `marketConfidence`, `assetValue`, and `workforceMorale`.
- The system can produce at least one delayed cascade where a prior choice worsens a later weather or maintenance shock.
- The board packet and consequence feed summarize operational issues in executive language.

Example:

- The player defers maintenance twice.
- Maintenance backlog rises.
- A weather front hits a fragile hub.
- The network cannot absorb disruption.
- Public anger and legal heat rise.
- Press or regulator factions gain leverage.
- A dossier thread may receive evidence if safety theater was active.

Acceptance criteria:

- at least three authored decisions modify operational state
- at least one hazard event reads operational fragility
- at least one board packet signal summarizes network stress
- tests cover weather, backlog, staffing, and route fragility interactions
- the UI does not expose detailed route controls

### 12.7 Dossier and scandal machine v1

History should become evidence.

`V0.5` should introduce dossier threads that accumulate evidence from decisions, events, faction intents, and operational incidents.

Required dossier themes:

- `insider_trading`
- `maintenance_fraud`
- `labor_abuse`
- `regulatory_capture`
- `offshore_evasion`

Recommended state:

```ts
type DossierTheme =
  | "insider_trading"
  | "maintenance_fraud"
  | "labor_abuse"
  | "regulatory_capture"
  | "offshore_evasion";

interface DossierThread {
  id: string;
  theme: DossierTheme;
  severity: number;
  evidenceWeight: number;
  witnesses: string[];
  linkedDecisionIds: string[];
  linkedEventIds: string[];

```

Requirements:

- Selected decisions can add evidence fragments.
- Events can add evidence fragments.
- Faction intents can amplify, surface, or suppress dossier threads.
- Dossier severity can influence legal heat, ending flavor, and post-run analysis.
- Dossiers should not create instant failure by surprise. They should create pressure and explainability.
- Two runs that reach the same ending should be able to produce different scandal recaps.

Acceptance criteria:

- at least three dossier themes can accumulate evidence in real runs
- at least one dossier theme can influence legal pressure
- ending recap includes top dossier threads and likely cause of exposure or escape
- tests cover evidence accumulation, dormancy, surfacing, and recap summaries

### 12.8 Post-run analytics and local retention

`V0.5` should make failed and successful runs more satisfying to read.

Requirements:

- Add a post-run recap that summarizes ending, major factions, top dossier threads, missed exit windows, and critical decision chains.
- Store enough local run summary data to revisit recent run outcomes.
- Keep storage local-first.
- Avoid turning the game into a meta-progression grind.
- Challenge seeds are allowed as a local deterministic feature.

Minimum recap sections:

- final ending
- rounds survived
- personal wealth versus company condition
- dominant faction adversary
- strongest dossier thread
- most damaging operational cascade
- missed exit window, if detectable
- top three selected decision chains

Acceptance criteria:

- ending screen has a richer recap for at least prison, extraction, merger, and Bahamas outcomes
- recent run summaries survive refresh
- run archive can be cleared safely
- Playwright covers reaching an ending and seeing recap content

## 13. UI and UX Requirements

Non-requirements:

- no large route map in the first operational implementation

## 16. Architecture Requirements

### 16.1 Determinism

Every system added in `V0.5` must be deterministic for:

- initial seed
- content version
- run state
- selected decision ids
- deterministic bot policy, when used by tools

Random-looking behavior must use the existing seeded random approach or a documented successor.

### 16.2 Pure systems

Simulation modules should be pure by default:

- explicit inputs
- explicit outputs
- no React imports
- no Zustand imports
- no DOM access
- no browser storage access
- no hidden singleton mutation except carefully documented content caches

### 16.3 Public interfaces

New systems should expose narrow public functions.

Example style:

```ts
export function resolveFactionLayer(input: {
  run: RunState;
  manifest: CompiledContentManifest;
  selectedDecisionIds: string[];
  emittedEventIds: string[];
}): FactionResolution {
  const grievances = collectFactionGrievances(input);
  const factions = updateFactionStates(input.run.factions, grievances);
  const intents = selectFactionIntents(factions, input.run);

  return {
    factions,
    intents,
    metricImpacts: deriveFactionMetricImpacts(intents),
    scheduledEvents: scheduleFactionEvents(intents, input.manifest),
    decisionModifiers: deriveFactionDecisionModifiers(intents),
    briefingSignals: summarizeFactionSignals(intents),
  };
}
```

Style expectations:

- prefer descriptive types over loosely shaped objects
- prefer discriminated unions for event, intent, diagnostic, and report variants
- keep tuning constants named and centralized
- write short comments only when they explain non-obvious design intent
- keep functions small enough to unit test

### 16.4 Resolution order

The `V0.5` resolution order must be documented and tested.

Recommended order:

1. validate selected decisions
2. apply selected decision direct impacts
3. collect evidence fragments from selected decisions
4. update operational state from selected decisions
5. update faction state from selected decisions and current metrics
6. schedule guaranteed consequences from selected decisions and faction intents
7. apply operating drift or causal influence effects
8. resolve operational quarter
9. resolve due scheduled events
10. evaluate hazard events
11. update dossiers from emitted events and faction intents
12. apply event impacts
13. check automatic endings
14. check authored exit endings
15. assemble board packet, history, tray memory, and run summary state

The final implementation may adjust this order, but the order must be written down in code comments or docs and covered by regression tests.

### 16.5 Performance and memory

Requirements:

- The live `RunState` should remain compact enough for frequent cloning in tests and tooling.
- Long text should be referenced by ids where practical.
- Visible history should remain bounded.
- Run archives should store summaries or deltas rather than full repeated prose.
- Long simulation reports must run outside React.
- If a worker is introduced, it should be used for analysis jobs before normal play depends on it.

## 20. Boundaries

Always:

- preserve deterministic behavior
- keep simulation rules outside React components
- update tests with behavior changes
- update docs with meaningful system changes
- run `npm run check`, `npm run content:validate`, and `npm run build` before marking implementation complete
- keep content fictionalized and tone-consistent
- document any balance threshold changes

Ask first:

- adding a new dependency
- migrating package manager or workspace structure
- adding a backend or cloud service
- changing stable product scope in `docs/PRD.md`
- removing an existing ending
- changing the visible metric vector
- replacing the current persistence strategy
- making route networks directly controllable by the player

Never:

- introduce runtime LLM behavior into the turn loop
- make the game require network access to play
- hide non-deterministic randomness inside simulation logic
- loosen tests to pass new behavior without documenting why
- commit secrets or generated dependency folders
- recreate real-world disasters as spectacle
- turn the UI into a raw developer diagnostics console

## 24. Risk Register

### Risk: Scope becomes too large to land

Mitigation:

- Implement in phases.
- Keep each phase shippable behind tests.
- Prefer vertical slices over half-built large systems.
- Defer full workspace migration, backend, and route-map UI.

### Risk: The game becomes obscure

Mitigation:

- Every new system must produce player-readable signals.
- Board packet and recap must explain cause, not just state.
- Avoid exposing raw internal meters unless they matter.

### Risk: New systems create random-feeling punishment

Mitigation:

- Use deterministic planners and hazard rules.
- Give warning signals before severe consequences.
- Tie dossier and faction escalation to repeated behavior.

### Risk: Balance reports become noisy

Mitigation:

- Keep reports concise by default.
- Emit artifacts for deep inspection.
- Separate hard failures from advisory warnings.

### Risk: Save migration becomes fragile

Mitigation:

- Add migration tests before changing live persistence.
- Keep a safe fallback for unsupported saves.
- Avoid storing heavy text in active run state.

### Risk: Operational layer drifts into route management

Mitigation:

- No manual flight assignment.
- No route editor in `V0.5`.
- Expose operational state as board intelligence and consequences.

### Risk: Faction AI becomes opaque

Mitigation:

- Use simple deterministic intent selection.
- Store recent grievances.
- Surface the top cause in board packet and recap.
