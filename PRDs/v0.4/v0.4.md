## 1. Summary

`V0.4` prepares **The Lank Forenzo Simulator** for a Hierarchical Multi-Agent System workflow.

The current game is already playable, data-driven, deterministic, and well documented. The next risk is not lack of features. The next risk is that five parallel agents begin changing shared contracts, simulation rules, content registries, persistence, UI surfaces, and tests without a clear chain of command.

This iteration should harden the architecture so future feature work can move faster without turning the project into a contested battlefield.

`V0.4` should:

- split the concentrated round-resolution module into explicit deterministic systems
- centralize shared rule evaluation, metric bounds, impact application, and ending checks
- formalize save payloads with schema validation and migrations
- reduce manual content-registration friction and improve content diagnostics
- add balance, reachability, and seeded-regression tooling
- document the five-platoon command structure for future execution

This is an architecture and tooling iteration. It is not a major content expansion or full UI redesign.

## 2. Reconnaissance Findings

The `V0.4` plan is based on the architectural reconnaissance completed on 2026-04-13.

### 2.1 Current strengths

- The repo already matches the intended clean browser architecture: app, screens, components, simulation, theme, content, docs, tests.
- The simulation is deterministic and can be tested outside React.
- Authored content is split into validated packs.
- The product currently has `100` decisions, `10` decision packs, `132` events, and `5` endings.
- The UI is composed from clear surfaces: board packet, metric rail, decision tray, and consequence feed.
- Metric polarity rules are centralized enough that inverse-pressure metrics read correctly.
- Existing quality gates are healthy: lint, typecheck, unit/component tests, content validation, e2e smoke, and production build all pass.

### 2.2 Current weaknesses

- `src/simulation/resolution/resolveRound.ts` carries too many responsibilities in one module.
- Requirement evaluation exists in both event and decision paths.
- Metric bounds, impact application, operating drift formulas, event processing, ending checks, and history generation are not yet cleanly separated.
- `src/simulation/systems/decisionEngine.ts` contains hard-coded scoring weights and tray policy.
- Save persistence exists through Zustand, but there is no explicit save schema or migration table.
- Content pack registration requires touching multiple files and enums.
- Tests prove the basic loop, but do not yet cover long-run balance, content reachability, save migration, or seeded campaign regressions.

## 3. Problem Statement

The project is now entering a scale phase.

The current architecture can support one careful developer. It is not yet prepared for five specialized agents working in parallel unless shared contracts become more explicit.

The core problem is coordination risk:

- Simulation work can collide with persistence and tests.
- Content work can collide with schemas and type unions.
- UI work can accidentally encode gameplay logic.
- QA work lacks enough harnesses to detect balance or reachability regressions.
- Future agents need to know which files are shared contracts and which files are safe local work zones.

`V0.4` must turn the current playable build into a command-ready architecture.

## 4. Goals

This iteration should:

- preserve the current game behavior unless a behavior change is explicitly documented
- make deterministic resolution order easier to inspect, test, and extend
- create shared primitives for requirements, impacts, metric bounds, flags, endings, and history
- make save payloads versioned, validated, and migratable
- improve content authoring safety without over-engineering a content management system
- add tooling that tells future agents whether decisions and events are reachable in seeded runs
- provide clear platoon boundaries for the HMAS workflow
- update docs in the same pass so future agents can re-enter quickly

## 5. Non-Goals

This iteration does not:

- add a new campaign, airline, executive identity, or major endgame path
- expand the decision library beyond opportunistic small fixtures needed for tests
- redesign the full run screen
- introduce a backend, cloud saves, accounts, analytics, or multiplayer
- change the product thesis or stable MVP scope
- replace Zustand, React Router, Vite, Vitest, Playwright, or the current theme-token approach
- convert the game into a procedural or LLM-driven narrative system

## 6. Design Stance

`V0.4` should be conservative, disciplined, and contract-first.

The correct result is not a glamorous rewrite. The correct result is a project that a five-agent battalion can improve without losing determinism, content integrity, or product identity.

Guiding principles:

- Keep the player-facing loop intact while internals become clearer.
- Prefer pure functions for simulation modules.
- Put shared rules in one place before extending them.
- Make save migration boring and explicit.
- Keep content authoring friendly to humans.
- Add tooling that produces actionable reports, not noisy dashboards.
- Treat docs as part of the operational chain of command.

## 8. Architecture Requirements

### 8.1 Resolution pipeline

Requirements:

- `resolveRound` should remain the public entry point for round advancement.
- Resolution order must stay deterministic and documented.
- Internals should be split into named steps that can be unit-tested independently.
- Each step should accept explicit inputs and return explicit outputs.
- No extracted system should import React, Zustand, browser storage, or UI modules.

Recommended step boundaries:

1. validate selected decisions against current run state
2. apply selected decision impacts
3. schedule delayed consequences
4. apply operating drift
5. resolve due delayed events
6. select and resolve ambient event
7. check automatic and authored endings
8. assemble history, flags, event counts, pending events, and next round state

Implementation guidance:

- Preserve behavior first; optimize formulas later.
- Use regression tests around existing representative runs before and after extraction.
- Keep the extracted modules small enough that future platoons can own them independently.

### 8.2 Shared rule engine

Requirements:

- Decision and event requirements must use one shared evaluator.
- The evaluator must support existing fields: `roundAtLeast`, `roundAtMost`, `metricMin`, `metricMax`, `flagsAll`, and `flagsNone`.
- The evaluator should be usable by decision curation, event resolution, validation tools, and future content diagnostics.
- Tests must cover boundary cases for each requirement type.

Implementation guidance:

- Do not create a broad generic rules DSL yet.
- Keep the contract close to the current content schema.
- If a future rules DSL becomes necessary, it should be a later versioned iteration.

### 8.3 Metric bounds and impacts

Requirements:

- Metric bounds should be declared in one canonical module.
- Impact application should use that canonical bounds module.
- Metric polarity and metric bounds should remain related but separate concepts.
- Tests must cover clamping behavior for every bounded metric and money-like metric.

Implementation guidance:

- Avoid scattering magic upper and lower bounds across simulation files.
- Keep existing values unless a balance change is explicitly approved.

### 8.4 Endings

Requirements:

- Automatic ending checks must be extracted from the main resolution function.
- Authored exit endings must remain supported through decision definitions.
- Tests must cover prison, forced removal, merger, extraction, and Bahamas paths.
- Ending rules should be easy to inspect from tests and docs.

Implementation guidance:

- Do not make endings data-driven unless it reduces real duplication.
- Preserve current readable TypeScript rules if that is simpler.

### 8.5 Persistence

Requirements:

- Save payloads must have an explicit version.
- Persisted run state must be validated with Zod before hydration.
- A migration table must exist even if `v1` only migrates to itself.
- Corrupt or unsupported saves must fail safely without breaking app boot.
- Theme persistence must keep working.
- Tests must cover valid save hydration, corrupt save fallback, and at least one example migration.

Implementation guidance:

- Prefer `localStorage` for this iteration unless IndexedDB becomes necessary for a concrete reason.
- Put storage contracts under `src/lib/storage/`.
- Keep persistence separate from simulation rules.

### 8.6 Content registry and diagnostics

Requirements:

- Adding a decision pack should require fewer coordinated edits than it does now.
- Validation must continue to catch duplicate IDs and broken delayed-consequence references.
- Validation should also report:
  - decision count by pack
  - decision count by group
  - event count by kind
  - unknown or orphan-like flags
  - delayed events that are never referenced
  - decisions whose requirements are likely impossible under current metric bounds
- The validation output should stay concise enough to be useful in CI.

Implementation guidance:

- Prefer a manifest or inferred registry over a full content pipeline.
- Do not introduce dynamic runtime fetches for local JSON content.

### 8.7 Balance and reachability tooling

Requirements:

- Add a deterministic script that can simulate many seeded runs without the browser.
- The script should report ending distribution, average run length, surfaced decision coverage, triggered event coverage, and repeated-tray pressure.
- The script should be suitable for local development and future CI gating.
- The report should help designers tune the game without reading raw test logs.

Implementation guidance:

- Start with a deterministic bot policy that chooses from available decisions using a simple, documented heuristic.
- Do not attempt to solve optimal play in this iteration.
- The goal is regression detection, not game theory perfection.

## 9. UX Requirements

`V0.4` is not a full UI redesign, but it should improve command clarity where architecture touches the UI.

Requirements:

- Decision cards should expose selected state accessibly.
- The resolve/end-turn action should be clear when no decisions are selected.
- The consequence feed should remain readable as history sources grow.
- Theme switching must continue to work across `Earth` and `Armonk Blue`.
- The primary run lifecycle should be covered by Playwright beyond only starting a run.

Implementation guidance:

- Keep UI changes narrow.
- Do not introduce new game screens unless needed for save error recovery or diagnostics.
- Preserve the board-packet, metric-rail, decision-tray, event-feed mental model.

## 14. Risks And Mitigations

| Risk                                                 | Impact | Mitigation                                                                      |
| ---------------------------------------------------- | ------ | ------------------------------------------------------------------------------- |
| Refactor changes game balance accidentally           | High   | Add regression tests before extraction and keep formulas unchanged              |
| Five platoons collide in shared files                | High   | Freeze shared contracts early and route changes through Platoon 1 or 3          |
| Save migration breaks existing local runs            | High   | Validate and migrate defensively; keep fallback behavior safe                   |
| Content diagnostics become noisy                     | Medium | Keep output concise and separate warnings from hard failures                    |
| Seeded bot produces misleading balance conclusions   | Medium | Document bot policy and treat report as regression signal, not perfect strategy |
| UI hardening expands into redesign                   | Medium | Restrict UI work to accessibility, lifecycle coverage, and narrow clarity fixes |
| Pack registry cleanup over-engineers content loading | Medium | Prefer a simple manifest; no runtime fetch pipeline in this version             |

## 15. Implementation Decisions

- Seeded campaign reports are generated locally through `npm run simulate:runs` / `npm run balance:report`; no snapshot artifact is committed in `V0.4`.
- Content validation treats broken references and required-only flags as hard errors, while advisory diagnostics such as unreferenced delayed events and set-only flags remain warnings.
- Save migration preserves the current pre-`V0.4` Zustand payload shape as a legacy version.
- Decision curation weights remain TypeScript constants, but they now live in an explicit scoring-policy object instead of being scattered as anonymous magic numbers.
