# V0.8 PRD: Phase-Based Round Flow

Status: documented direction for active implementation
Owner: BinaryOutlook
Last updated: 2026-05-12
Iteration type: run-screen UX direction, responsive interaction model, and implementation coordination

Design reference: [`v0.8-demo.html`](v0.8-demo.html)

## Summary

`V0.8` records the revised round-flow UX direction for **The Lank Forenzo Simulator**. The product is moving away from the earlier assumption that the entire round must live inside one strict, scroll-less, always-visible surface. That constraint helped when the game had fewer systems, but it now compresses too much state, too many controls, and too much decision comparison into one frame.

The new direction keeps the severe boardroom identity, the local-first browser shell, and deterministic simulation intact while making the round feel more like a guided sequence:

1. **Read / Board Packet**: inspect the quarter, pressure reads, state, reserves, and consequence feed.
2. **Choose Plays**: compare and select decisions in a dedicated selection surface.
3. **Resolve / End Round**: validate selections, confirm the chosen plays, and advance the simulation.

The philosophy is simple: give each kind of thinking its own space. The player should not have to compare five dense executive plays while every metric, warning, feed item, and resolve affordance fights for the same few pixels.

## Problem

The current round screen carries more responsibility than the original fitted layout was designed to hold:

- board-packet context
- company and personal state
- pressure reads and metric direction
- strategic reserve affordability
- consequence feed
- five dense decision cards
- selected-count validation
- end-round control
- responsive portrait controls

The resulting strain can be described as:

$$
\text{interaction strain} \propto \frac{\text{state} + \text{choices} + \text{click targets}}{\text{breathing room}}
$$

The product should still feel like a command surface, not a blog page. But strict one-frame compression is no longer the right proxy for quality.

## Goals

1. Establish a durable phase-based mental model for a round.
2. Explain why the strict scroll-less single-surface rule is being relaxed.
3. Define the expected role of the decision-selection phase.
4. Define the expected role of the end-round confirmation and incomplete-selection recovery modal.
5. Give responsive implementation guidance that preserves a fitted app-shell feel without forcing cramped decision comparison.
6. Coordinate the implementation issues so agents do not re-litigate the same UX direction in separate PRs.

## Non-Goals

- Do not change simulation formulas, decision outcomes, scheduler behavior, save format, endings, or balance.
- Do not introduce a backend, account system, cloud save, or mobile-native app shell.
- Do not require one specific technical implementation such as routes versus internal component state.
- Do not make the confirmation modal the place where players primarily compare available decisions.
- Do not create a second mobile-only ruleset or device-specific decision availability.

## Round-Flow Philosophy

### Phase 1: Read / Board Packet

The Read phase is for understanding the quarter before committing to action.

It should foreground:

- board-packet summary and current round frame
- major company and personal metrics
- pressure reads and metric direction
- strategic reserve balances
- recent consequence feed and relevant history
- any hazards, faction pressure, or dossier signals the player should account for before choosing plays

This phase answers: **What situation am I walking into?**

### Phase 2: Choose Plays

The Choose Plays phase is the primary decision surface.

It should foreground:

- all available decisions for the round
- title, category, summary, cost, and consequence preview for each decision
- selected and unselected states
- affordability and reserve shortfalls
- a persistent selection counter such as `0/2 selected`
- mouse, touch, keyboard, and focus affordances for selection and deselection

This phase answers: **Which ugly moves am I willing to own?**

The player should be able to compare the five available plays without cards clipping important chips or competing with the full board packet. Narrow screens may stack decisions vertically if the phase controls remain obvious.

### Phase 3: Resolve / End Round

The Resolve phase is a confirmation and recovery layer.

When selections are complete, it should:

- summarize the selected decisions
- confirm that advancing will resolve the round
- keep copy concise and consistent with the tense institutional tone
- let the player cancel without losing selected decisions

When selections are incomplete, it should:

- explain how many choices are still required
- provide a direct path back to Choose Plays
- preserve all current selections
- avoid burying the player in another full decision-comparison surface

This phase answers: **Am I ready to seal this quarter?**

## End-Round Modal Role

The modal is a safety net and final review point.

It is not a replacement for the dedicated decision-selection phase.

Expected behavior:

- Complete selection: show a short selected-play summary and resolve confirmation.
- Incomplete selection: show the missing count and a direct recovery action.
- Dismissal: return to the current round without clearing selected decisions.
- Accessibility: support predictable escape, focus return, and keyboard confirmation behavior.

If the modal grows large enough that the player is comparing all five unselected cards inside it, the implementation has drifted away from this direction.

## Responsive Expectations

The earlier fitted-layout work remains useful. The revised direction relaxes the absolute scroll-less rule without abandoning the app-like shell.

Responsive implementations should:

- keep phase navigation, selection count, and resolve affordances easy to find
- preserve readable decision cost and consequence chips
- avoid horizontal scrolling
- respect safe-area and dynamic-viewport constraints
- allow vertical breathing room inside the Choose Plays phase when comparison needs it
- keep the same simulation data and decision rules across desktop, tablet, and phone viewports
- retain the severe, ledger-like, institutional visual identity

Preferred narrow-screen behavior is a clear phase surface with internal or page-level breathing room, not a cramped facsimile of the old all-at-once layout.

Agents working on #69, #70, and #71 should treat this packet as the common UX contract. They may choose the smallest compatible technical shape, but they should not reintroduce the old rule that every major round surface must remain simultaneously visible.

## Technical Guidance

The implementation may use:

- a `roundPhase` state inside the run screen
- nested route-like views under the existing run route
- a phase component that controls which panels are emphasized
- a modal dialog for final confirmation and recovery

The chosen shape should preserve these boundaries:

- Simulation rules remain in pure TypeScript simulation modules.
- React components render run state and dispatch existing selection or resolution actions.
- Phase transitions do not mutate simulation state except when the player confirms round resolution.
- Decision selection continues to use the existing deterministic decision tray and selection contracts.
- Tests should prove that UX restructuring does not change deterministic outcomes.

## Acceptance Criteria

- The stable PRD layer explains why strict scroll-less single-surface compression is being relaxed.
- Future agents can name the intended phases: Read / Board Packet, Choose Plays, Resolve / End Round.
- The decision-selection phase is documented as the primary comparison and selection surface.
- The end-round modal is documented as confirmation and recovery, not the primary choice surface.
- Responsive expectations are clear enough to guide both desktop and narrow viewport implementations.
- README points contributors to this packet when they need the current round-flow direction.
