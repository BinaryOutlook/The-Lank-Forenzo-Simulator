# V0.2 PRD

Status: Implemented
Owner: BinaryOutlook
Last updated: 2026-04-09
Iteration type: Content scale and indicator semantics pass

## 1. Summary

`V0.2` is a systems-and-content expansion pass for the run screen.

This packet addresses three visible weaknesses in the current build:

- the event library is too small to sustain variety across repeated runs
- several indicators still rely on an overly simple assumption that positive numbers are good and negative numbers are bad
- the right-hand consequence feed keeps too much history on screen at once

This iteration is about scale, semantic clarity, and scan discipline.

It should make the simulation feel less repetitive, make metric direction easier to interpret correctly, and keep the run screen's right rail focused on the latest fallout.

## 2. Problem Statement

The current run loop is playable, but three structural issues are now limiting the experience:

- The event pool is still tiny. The current baseline is `18` total events in a single file, split across `12` delayed events and `6` ambient events. Even with weighting and seen penalties, repetition arrives too quickly.
- Some indicators still behave as if numeric sign alone defines whether a change is good or bad. That assumption breaks down for inverse-pressure metrics such as `legalHeat` and `publicAnger`, where lower values are healthier.
- The right-hand consequence feed currently renders the full stored history window, which makes the rail taller, noisier, and less useful as a quick "what just happened?" read.

Together, these issues reduce replay variety, weaken decision readability, and make the run screen harder to scan.

## 3. Goals

This iteration should:

- expand the authored event surface well beyond the current `18`-event baseline
- restructure event authoring so scaling toward roughly `30x` to `50x` the current event volume is maintainable
- define explicit polarity rules for every stat indicator and ranking indicator instead of inferring meaning from raw sign alone
- ensure inverse-pressure metrics such as `legalHeat` and `publicAnger` no longer communicate backwards
- limit the visible consequence feed on the run screen to the `10` most recent items
- preserve the product's existing tone, theme identity, and deterministic local-first architecture

## 4. Non-Goals

This iteration does not:

- introduce a full narrative campaign or branching story engine
- redesign the whole run-screen layout
- replace the current round structure
- add backend services or live content delivery
- solve every balance issue in the simulation
- require a fully new visual system beyond the indicator and feed changes needed here

## 6. Requirements

### 6.1 Event library scale-up

The event system must stop feeling content-starved.

Requirements:

- The current `18`-event baseline must be treated as insufficient.
- `V0.2` should materially increase the authored event count and establish a content structure that can reasonably grow toward roughly `540` to `900` total events over time.
- Event variety must improve across both `ambient` and `delayed` categories.
- The authored pool should cover more combinations of labor, finance, operations, market, legal, safety, board, press, personal, merger, and offshore pressure.
- Repeated runs should surface meaningfully different event mixes before obvious repetition sets in.
- Content organization should no longer assume a single flat event file is the only maintainable shape if splitting by pack, category, or phase improves authoring clarity.

Implementation guidance:

- Prefer a content structure that makes large authored pools easy to review and validate.
- Preserve deterministic selection behavior and seen-penalty logic unless a better repeat-suppression mechanism is introduced deliberately.
- If the full `30x` to `50x` target is not practical in one implementation pass, the structure and workflow must still clearly support that destination rather than locking the repo into another tiny-pool ceiling.

### 6.2 Metric polarity audit and systemization

The product must stop assuming that positive deltas are universally good and negative deltas are universally bad.

Requirements:

- Every stat indicator and ranking indicator must have an explicit reviewed direction.
- Each metric should be classified intentionally, for example as `higher-is-better`, `lower-is-better`, or `contextual`, rather than being judged by delta sign alone.
- Inverse-pressure metrics such as `legalHeat` and `publicAnger` must communicate correctly when they move downward.
- Any component that colors, ranks, scores, or summarizes metrics must use reviewed metric semantics instead of hard-coded sign assumptions where those assumptions can mislead.
- The review must cover all existing run metrics, including `airlineCash`, `personalWealth`, `debt`, `assetValue`, `workforceSize`, `workforceMorale`, `marketConfidence`, `creditorPatience`, `legalHeat`, `safetyIntegrity`, `publicAnger`, `stockPrice`, and `offshoreReadiness`.
- The final reviewed polarity map should be documented in code or validation-friendly metadata, not left implicit across multiple components.

Implementation guidance:

- A shared metric-semantics definition is preferred over duplicating "good" versus "bad" logic in each component.
- Metrics that are strategically ambiguous should be treated as contextual instead of being forced into fake green/red certainty.
- Review decision previews, metric rails, history tone, briefing signals, and any ranking or summary logic that currently embeds directional judgment.

### 6.3 Consequence-feed cap

The consequence feed on the right-hand side must become a tighter recent-history view.

Requirements:

- The run-screen consequence feed must render no more than `10` entries at a time.
- The feed should continue to prioritize the newest items first.
- The cap should improve scan speed without making the feed feel broken or arbitrarily truncated.
- Internal history retention may remain deeper than `10` if the simulation needs it elsewhere, but the right-rail presentation must stop at `10`.

Implementation guidance:

- Solve this as a display rule first, not a destructive loss of useful state unless a state cap change is chosen on purpose.
- Preserve stable ordering and source labeling so the last `10` entries read as a coherent chain of consequences.

## 7. UX Acceptance Criteria

The iteration is successful when all of the following are true:

- The event system no longer feels constrained to a tiny, quickly repeating pool.
- Event authoring is organized in a way that clearly supports continued growth beyond the initial `V0.2` expansion.
- A reviewed metric-direction definition exists for every current run metric.
- Indicators no longer communicate inverse-pressure metrics backwards simply because a number is negative or positive.
- The right-hand consequence feed shows at most `10` entries in the run screen.
- The run screen remains readable and thematically consistent in both `Earth` and `Armonk Blue`.

## 8. QA Notes

Check these scenarios before closing the iteration:

- multiple fresh runs to confirm improved event variety
- longer runs to confirm delayed and ambient events do not repeat too aggressively
- metrics where lower is healthier, especially `legalHeat` and `publicAnger`
- metrics where direction may be contextual rather than purely good or bad
- both themes on the run screen
- rounds where the run history exceeds `10` entries to confirm the feed cap behaves correctly
- content validation after any event-file restructuring

## 9. Open Questions

- Whether `V0.2` should fully deliver the long-term `30x` to `50x` event target or establish the scalable structure plus the first major expansion should be decided before implementation starts.
- Some metrics may not belong to a simple higher-is-better or lower-is-better model. `workforceSize` is the clearest candidate for a contextual rule and should be reviewed carefully.
- If a deeper history log is useful later, the product may want a dedicated full-log surface instead of using the right rail as both archive and recap.
