# V0.9 PRD: Local Save Load Manager

Status: implemented first pass
Owner: BinaryOutlook
Last updated: 2026-05-13
Iteration type: persistence UX, non-encrypted save boundary, local import/export

Design reference: [`v0.9-demo.html`](v0.9-demo.html)

## Summary

`V0.9` adds the first dedicated Load Manager for **The Lank Forenzo Simulator**.
The work turns local persistence from invisible autosave plumbing into a clear
player-facing surface with browser save slots, plain JSON file import, and save
export.

The first pass is intentionally non-encrypted. Its job is to stabilize the
payload boundary, player flow, and migration assumptions before encryption or
tamper resistance is introduced.

## Goals

1. Add a dedicated `/load` Load Manager screen.
2. Let players load saved browser sessions.
3. Let players load compatible local save files.
4. Let players manually create browser save slots.
5. Let players export the current payload as a local JSON file.
6. Record the save format, migration assumptions, and tamper policy.
7. Create follow-up planning for encrypted or tamper-resistant saves.

## Non-Goals

- Do not add encryption in this pass.
- Do not add cloud saves, accounts, sync, or multiplayer persistence.
- Do not promise anti-cheat or save-file authenticity.
- Do not persist transient UI state such as scroll position, active phase, or
  modal focus.

## Product Behavior

The Load Manager presents three options:

1. **Load saved sessions**: list manual browser slots and load or delete them.
2. **Load from local files**: import compatible `.tlfs-save.json` or legacy save
   JSON.
3. **Save functionality**: label the current payload, save it to a browser slot,
   or export it as a file.

Entry points should be visible from the landing page, Options page, active run
screen, and decision-selection phase.

## Save Boundary

The durable save payload is:

```ts
interface GameSavePayload {
  theme: ThemeName;
  settings: GameSettings;
  run: RunState | null;
}
```

The boundary deliberately follows product state, not UI state. This keeps saves
portable across layout changes while preserving everything needed to resume a
run.

## Migration And Failure Rules

- Save payloads use `GAME_SAVE_STORAGE_VERSION`.
- File exports use a `fileVersion` wrapper around the payload.
- Older compatible payloads are migrated or coerced through the storage layer.
- Corrupt JSON fails with a visible error.
- Future file-format or save-schema versions are rejected during explicit import.
- Browser save slots are capped so local storage does not grow without bound.

## Acceptance Criteria

- The app has a dedicated Load Manager route.
- Landing, Options, and gameplay surfaces expose a Load Manager action.
- A player can create and load a browser save slot.
- A player can import and export a non-encrypted local JSON save file.
- The save boundary and migration assumptions are documented.
- Encrypted save work is separated into a follow-up ticket.
